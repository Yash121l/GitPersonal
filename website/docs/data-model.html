Persistence Model

Forge splits state deliberately between PostgreSQL metadata and Git-native filesystem storage.

This project does not force everything into one database and it does not pretend the filesystem is only an implementation detail. A Git host has two truths: metadata in a relational store and repositories as Git-native object databases on disk. Forge keeps those roles explicit.

Storage Boundaries

Layer What lives there
PostgreSQL Users, sessions, repositories, SSH keys, and maintenance metadata.
/data/repos Sharded bare Git repositories, staging directories, and trash directories.
/data/ssh Generated or persisted SSH host key material for the Forge SSH listener.
In-memory store Test and no-database fallback only; not a durable production backend.

Current Schema

Table Purpose
users Core identity rows. Stores username, bcrypt password hash, role, and profile placeholders.
sessions Revocable session rows keyed by token identifier with `expires_at` and optional `revoked_at`.
repositories Repository metadata, ownership, visibility, default branch, repo path, size, and maintenance timestamps.
ssh_keys Stored public keys with SHA-256 fingerprint uniqueness and `last_used_at` tracking.
organizations Reserved for future org ownership. Present in schema, not yet exposed through the API.
org_members Reserved for future org membership and role mapping.

Migration `000002_repository_maintenance` adds `last_indexed_at` and `last_maintained_at` to the `repositories` table so background maintenance can leave a durable trace.

Repository Invariants

  • Every repository row must be owned by exactly one owner type: either `owner_user_id` or `owner_org_id`, but not both.
  • User-owned repository names are unique per owner case-insensitively through a partial unique index on `lower(name)`.
  • Organization-owned repository names have the same case-insensitive uniqueness rule through a separate partial unique index.
  • `repo_path` stores the actual filesystem path used by Git commands; the application does not recompute it from metadata at request time.
  • The current create flow only provisions user-owned repositories even though the schema is already shaped for organizations.

Filesystem paths are derived from a SHA-256 fanout of `owner/name`, using two one-byte hexadecimal prefix directories before the owner and repository slug. This keeps large repository sets from collapsing into a single hot directory.

Store Interface

The `internal/store` package defines the application persistence contract. That contract includes:

  • User creation and lookup.
  • Repository create, lookup, list, delete, and stat updates.
  • Session create, lookup by token identifier, and revocation.
  • SSH key create, lookup by fingerprint, and `last_used_at` updates.
  • A repository lease hook, `WithRepositoryLease`, for mutation serialization.
  • A `Check` method for readiness probes.

This split is intentional. The HTTP server and repository service code depend on the interface, not on PostgreSQL directly, which makes transport tests faster and keeps fallback behavior available.

Store Implementations

Implementation What it does today
internal/store/memory Uses Go maps and mutexes, including a per-repo mutex map for `WithRepositoryLease`. Ideal for tests and local no-DB runs.
internal/store/postgres Persists users, sessions, repos, and keys in PostgreSQL. It is the real deployment backend, but repository leasing is not yet implemented there.

There is also a helper `RepositoryLeaseKey` built from normalized `owner/name` using FNV-1a hashing. That makes the intended lock identity explicit even though the PostgreSQL implementation has not yet turned it into an advisory lock or other cross-process lease.

Current Gaps

  • Organizations exist in the schema but have no HTTP routes, authorization model, or repository ownership flow yet.
  • The repository authorization model lives mostly in application code and is currently simple: public read or owner write.
  • The PostgreSQL store does not yet serialize repository mutations across processes.
  • No audit tables, webhook tables, CI tables, or issue/PR tables are present in the live implementation yet.
  • Redis is present in deployment but not yet part of the active persistence or queue path.
