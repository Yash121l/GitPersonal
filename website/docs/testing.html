Quality And Verification

Forge is moving toward test-driven development with transport-level realism.

The project now includes more than handler-level unit checks. It exercises Git transport behavior, session revocation, repository maintenance, and local runtime deployment. That is the right direction for a system that claims to explain how a Git host works.

TDD Direction

“Test-driven going forward” in this project means new architectural claims should be backed by executable proof. If the system says it supports smart HTTP or SSH Git access, there should be tests that actually perform those actions against the code.

The tests are not purely mocked. They use real Git commands where that adds confidence. This is especially important for transport, because Git protocol bugs often hide behind clean-looking unit tests.

Current Automated Suite

Area What is covered now
Config Validation behavior, especially unsafe production defaults.
Repository service Provisioning, deletion, sharded paths, and maintenance stat updates.
HTTP server Auth lifecycle, repository CRUD, headers, readiness, and unauthorized access behavior.
Transport integration Session logout revocation, smart HTTP push/read, and SSH `ls-remote` with a registered key.

Transport Coverage

The most valuable tests currently in the codebase are the transport integration tests. They prove that:

  • A user can authenticate and lose access after logout because the session is actually revoked.
  • A repository can be created, pushed to over smart HTTP, and read back with `ls-remote`.
  • An SSH public key can be registered and then used to access the same repository over SSH.
  • Repository maintenance updates metadata after work is queued.

These tests are important because they validate system seams, not just internal function signatures.

Local Runtime Verification

Automated tests are necessary, but the project also now includes manual runtime verification against the real `docker-compose` stack. That live verification is what caught two important deployment bugs:

  • The first runtime image did not actually include `git-http-backend`, causing live smart HTTP to fail.
  • The non-root container needed an entrypoint strategy to repair mounted volume ownership before serving.

That is exactly why local deployment verification matters. A green unit suite alone would not have exposed those packaging and permission failures.

Next Test Targets

  • Authorization matrix tests once collaborators and team permissions exist.
  • Rate limiting and CSRF tests when those layers are added.
  • Webhook delivery worker tests once background jobs beyond maintenance are introduced.
  • Database-backed transport tests against PostgreSQL specifically, not only the in-memory store.
  • Upgrade-path tests for migration compatibility as the schema grows.
