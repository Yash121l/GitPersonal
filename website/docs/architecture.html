Implementation Details

Forge is a Git transport and metadata system built from explicit subsystems.

The architectural goal is clarity. Git repositories live on disk. PostgreSQL holds metadata and revocable sessions. HTTP and SSH transports map authenticated users to real Git server commands. Maintenance work is moved off the foreground request path.

This is intentionally a small-system architecture. It is suitable for a student learning project, a personal lab, or a small internal install. The architecture favors inspectability and correctness over distribution or multi-node complexity.

Storage Model

Forge uses bare repositories on disk. That choice is deliberate. Bare repositories are the correct Git server primitive because Git’s own transport commands already know how to read and write them. The sustainability question is not “bare repo or not”; it is how the surrounding system manages provisioning, concurrency, path layout, and maintenance.

Repository Layout

Repository paths are sharded to avoid a giant single directory. The current layout fans out by two hash-derived prefix directories before the owner and repository name: 

/data/repos/
  bf/35/yash/forge.git
  9a/10/team/demo.git

Provisioning Strategy

  • Create in a staging area first.
  • Run `git init --bare --shared=group`.
  • Write the `HEAD` ref explicitly for the configured default branch.
  • Apply repository config such as commit-graph and bitmap-oriented settings.
  • Move into the final path atomically.

Repository config written during provisioning

  • `core.sharedRepository=group` to allow cooperative access on shared volumes.
  • `gc.auto=0` and `receive.autogc=false` so maintenance is not triggered inline during user traffic.
  • `core.logAllRefUpdates=true` for ref update tracking inside the bare repository.
  • `repack.writeBitmaps=true`, `fetch.writeCommitGraph=true`, and `commitGraph.generationVersion=2` to bias the repository toward faster read traversal once maintenance runs.
  • `pack.useSparse=true` as a sensible default for packed object access.

Deletion is also staged. A repository is moved to a trash path first so metadata deletion and filesystem deletion are not coupled in a fragile all-or-nothing direct remove.

Path safety is explicit: filesystem operations reject paths outside the managed repository root, and the provisioner refuses to operate directly on the root path itself.

Authentication And Sessions

Forge uses bcrypt for passwords and JWT cookies for browser sessions, but the JWT is not treated as the only truth source. Each token gets a generated ID stored in the database. Logout revokes that session row, and the middleware checks the stored session on every authenticated request.

This matters because it gives the system a real revocation primitive. Without it, logout is only client-side cookie deletion and any stolen token remains valid until expiry.

What the store persists

  • Users
  • Sessions keyed by JWT `jti`
  • SSH public keys keyed by SHA-256 fingerprint
  • Repository metadata including repository size and maintenance timestamps

Cookie behavior is simple but deliberate. The cookie is `HttpOnly`, `SameSite=Lax`, scoped to `/`, and marked `Secure` when `FORGE_BASE_URL` uses HTTPS.

Smart HTTP Git Transport

Git HTTP traffic enters through /git/{owner}/{repo}.git paths. Forge resolves the repository, checks whether the request is read or write oriented, authenticates the user via session cookie or HTTP Basic credentials, and then invokes git-http-backend with CGI-style environment variables.

Request Flow

  1. Parse the incoming Git path into owner, repository, and suffix.
  2. Load repository metadata from the store.
  3. Determine whether the request is a fetch/list or a push/write.
  4. Authorize based on repository visibility and owner access.
  5. Invoke the backend binary with `GIT_PROJECT_ROOT`, `PATH_INFO`, `QUERY_STRING`, and request metadata.
  6. Relay CGI response headers and body back to the Git client.
  7. If the request wrote data, enqueue repository maintenance.

A notable implementation detail: the runtime image was switched so the deployed container actually ships the backend binary needed for smart HTTP. That was caught during live runtime verification, not only by unit tests.

Git-over-HTTP auth supports two paths: a browser session cookie or HTTP Basic credentials. That makes the same system usable from both browser clients and standard Git tooling without inventing a separate auth layer.

SSH Git Transport

Forge also exposes Git over SSH on a dedicated TCP listener. The SSH server validates public keys against the stored fingerprint table, then allows only Git-oriented commands. It does not provide a shell.

Accepted commands

  • git-upload-pack owner/repo.git for fetch and clone
  • git-receive-pack owner/repo.git for push

The SSH handler maps the authenticated key back to a stored user and then applies the same owner/public access model used by smart HTTP. Successful write operations enqueue maintenance work.

The SSH host key path is configured explicitly. If the key file does not exist, Forge creates an Ed25519 host key on startup, writes it to disk with `0600` permissions, and reuses it on later boots.

Maintenance And Indexing

Maintenance is intentionally backgrounded. User-facing requests should not block on heavyweight Git housekeeping. Forge currently enqueues repository maintenance after create and after successful write operations, then runs:

  • `git gc --auto`
  • `git commit-graph write --reachable`
  • filesystem size accounting for repository metadata

The store is updated with size_bytes, last_indexed_at, and last_maintained_at. This is the beginning of a real operational story, even though richer indexing and post-receive pipelines are still future work.

Queue semantics are intentionally lightweight. The scheduler keeps a bounded in-process queue and deduplicates pending work per `owner/repo` key so a burst of pushes does not linearly explode maintenance jobs. Each run executes with a 30-second timeout.

Runtime Boot And Failure Model

Startup is fail-fast. Config is validated before the process begins serving. PostgreSQL is opened and migrated before the HTTP or SSH servers listen. Repository storage is checked during service setup.

The deployment flow also contains a pragmatic container detail: because mounted volumes may be root-owned, the container entrypoint repairs ownership and then drops privileges to the non-root forge user. This was required to make local self-hosting work reliably through `docker-compose`.

  1. Load and validate config.
  2. Open the selected store backend.
  3. Apply embedded migrations when PostgreSQL is enabled.
  4. Initialize repository storage and start the maintenance worker.
  5. Build the HTTP router and, when enabled, the SSH listener.
  6. Start serving and wait for `SIGINT` or `SIGTERM` for graceful shutdown.

Current Limits

  • Authorization is still simple: owner access and public-read visibility only.
  • No web UI is built yet, so operational access is API and Git-client driven.
  • No webhook execution or CI runtime is wired in.
  • No multi-node coordination or distributed locking exists.
  • The PostgreSQL store currently does not implement a cross-process repository lease; `WithRepositoryLease` is effectively a no-op there.
  • Rate limiting, CSRF protection, and audit logging remain future hardening work.
