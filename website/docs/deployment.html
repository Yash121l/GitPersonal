Self-Hosting Guide

Forge is written to be run by the person learning from it.

The deployment story is intentionally local-first. You can run it on your own machine, on an internal lab server, or on a small VPS. The project is meant to teach the deployment edges of a Git host, not outsource them away.

Self-Hosting Stance

Forge takes a clear position: self-hosting is not a compromise tier. For a student or small trusted group, owning the deployment path is part of learning how the platform works. The project is documented to make that ownership practical.

Quick Start

  1. Copy `.env.example` to `.env` if you are starting from a fresh clone, then adjust values as needed.
  2. Run docker-compose up --build -d.
  3. Check docker-compose ps until `forge` is healthy.
  4. Verify http://127.0.0.1:3000/readyz.
  5. Use HTTP Git on port 3000 and SSH Git on port 2222.
docker-compose up --build -d
docker-compose ps
curl http://127.0.0.1:3000/readyz

GitHub Pages For The Documentation Site

The static website in website/ can also be published through GitHub Pages. The repository includes a GitHub Actions workflow that uploads the website/ directory as the Pages artifact and deploys it on pushes to main.

To use it, enable GitHub Pages in the repository settings and set the source to GitHub Actions. No build step is required because the site is plain HTML, CSS, and JavaScript.

Ports And Volumes

  • 3000: HTTP API and smart HTTP Git transport
  • 2222: SSH Git transport
  • 5432: PostgreSQL
  • 6379: Redis
  • 1025 / 8025: Mailpit SMTP and web UI

Data is persisted in named Docker volumes. The Forge container entrypoint repairs ownership of the mounted `/data` tree before dropping privileges to the non-root runtime user.

Operations Notes

  • Health uses /readyz, not only /healthz, because repository storage and database access both matter.
  • Database migrations are embedded and applied automatically at startup.
  • The container runtime was adjusted so `git-http-backend` exists in the image and smart HTTP works live.
  • Repository maintenance is scheduled after create/write operations rather than performed inline with user traffic.
  • Production mode enforces stronger config validation, including HTTPS base URL and a non-trivial secret.

Troubleshooting

Symptom Likely cause
Container restarts on boot Check `docker-compose logs forge`; most likely a permission or config validation failure.
Smart HTTP returns 502 Verify the runtime image contains `git-http-backend` and that the repository path exists under `/data/repos`.
SSH auth fails Confirm the public key was registered through `POST /api/v1/keys` and use the correct SSH username.
Ready endpoint fails Either PostgreSQL is unavailable or repository storage is not writable.
